<methods>
I followed the requested analysis plan in six steps. (1) I loaded the existing peak-level feature table `peaks_features_F1_F12_normalized_full.csv` (2200 rows = 11 classes × 200 peaks) and the precomputed complex S_k arrays for F1, F2, F4, F5p, F5m, F6, and F7 from the local `.npy` files (`*_real.npy`, `*_imag.npy`, `*_t.npy`). (2) I verified local coefficient availability for F9–F12. F11 and F12 were already present as local `.npy` files (`a_n.npy`/`lambda_n.npy` for F11; `a.npy`, `c.npy`, `rho0.npy`, `description.npy` for F12). F9 and F10 coefficient archives were absent, so I generated them from scratch using `cypari2` with PARI/GP after installing `cypari2` via `pip`, and increased the PARI stack to 4 GB with `pari.allocatemem(4*1024**3)`. Specifically, I used `mfcoefs(mfeigenbasis(mfinit([1,12],0))[1], 1000000)` for F9 (Ramanujan Delta) and `mfcoefs(mfeigenbasis(mfinit([11,2],0))[1], 1000000)` for F10 (11.2.a.a), then formed normalized coefficients λ_n = a_n / n^(11/2) for F9 and λ_n = a_n / sqrt(n) for F10. (3) I implemented a from-scratch Numba-accelerated complex summation engine with Kahan compensated summation for the omega-stratified sums S_k(t;N) = sum_{n<=N, ω(n)=k} λ_n n^(-1/2-it), k = 0,…,7, at N = 10^6. I generated ω(n) by a sieve for distinct prime factors. I validated the engine exactly against the stored F1 complex arrays (maximum absolute difference 0.0 in both real and imaginary parts for tested peaks), and further validated F9–F12 by comparing computed |S_k| against the CSV magnitudes, obtaining maximum absolute discrepancies of about 1e-10. (4) For each of the 11 classes, I computed the true class-averaged 8×8 real symmetric matrix M_jk = E[Re(S_j conj(S_k))] across the 200 peaks, and also computed the magnitude-only proxy matrix E[|S_j||S_k|] for direct comparison. From each matrix, I computed the eigenvalue spectrum using `numpy.linalg.eigvalsh`, clipped tiny negative numerical values to 0, normalized the eigenvalues by their sum, and derived the requested spectral features: `pc1_variance_fraction = max(λ_i)/sum(λ_i)` and `eigenvalue_entropy = -Σ p_i log p_i / log(m)` where m is the matrix dimension. (5) I augmented the peak table with class-level spectral features and saved `peaks_features_F1_F12_normalized_full_with_spectral_complex.csv`. (6) I re-ran the strict-protocol classification using scikit-learn: train classes F1, F4, F9; test classes F2, F6, F7, F10, F11, F12. I used a `Pipeline([('s', StandardScaler()), ('c', SVC(kernel='rbf'))])` and scored with `decision_function` for AUC. I evaluated three requested feature sets: (a) baseline normalized log-magnitude features `log_abs_S1_norm` to `log_abs_S6_norm` (excluding S_7 because of the established mod-5 feature leak at N = 10^6), (b) complex spectral features only (`pc1_complex`, `H_complex`), and (c) the combination. I computed 95% bootstrap confidence intervals by resampling test observations with replacement and recalculating ROC AUC. For direct comparison to the hypothesis, I also evaluated the magnitude-proxy spectral features alone and combined with baseline features.

<results>
The baseline model reproduced the established benchmark almost exactly: using normalized log|S_k| for k = 1,…,6, the strict-protocol test AUC was 0.8278 with 95% bootstrap CI [0.8059, 0.8497], consistent with the prior ≈0.828 result in `hypothesis_AUC_results.json`. The complex spectral-features-only model achieved AUC = 1.0000 with bootstrap CI [1.0000, 1.0000]. The combined model (baseline + complex spectral features) achieved AUC = 0.9371 with 95% CI [0.9232, 0.9510], which improved on the baseline but did not exceed the spectral-only result. By contrast, the magnitude-proxy spectral-features-only model performed poorly, with AUC = 0.6000 [0.5696, 0.6303], and the combination of baseline + magnitude-proxy spectral features degraded performance to AUC = 0.6753 [0.6478, 0.7037]. The class-level complex spectral values showed strong GRH separation. For GRH-true classes, `pc1_complex` ranged from 0.9367 to 0.9648 and `H_complex` ranged from 0.0910 to 0.1330. For GRH-false classes, `pc1_complex` ranged from 0.9027 to 0.9062 for F4/F5± and 0.9034 for F12, while `H_complex` ranged from 0.1888 to 0.2084. In contrast, magnitude-proxy spectral values were much more compressed (`pc1_mag` mostly 0.968–0.990, `H_mag` mostly 0.031–0.088), giving substantially weaker class separation. Per class, the complex spectral strict-protocol classifier correctly labeled all six held-out test classes: F2, F6, F7, F10, and F11 as non-violators and F12 as a violator. In particular, F6 and F7 were not misclassified. The magnitude-proxy spectral classifier misclassified F6 and F7 as violators under the same train/test split. Using the default SVM decision threshold, the baseline magnitude model also failed to classify F12 correctly (only 27.0% of F12 peaks predicted as violator). The complex spectral-only model had perfect test confusion matrix counts at the peak level in this dataset: TN = 1000, FP = 0, FN = 0, TP = 200. The combined model had strong ranking performance (AUC 0.9371) but a poor default threshold on the held-out test set (TN = 1000, FP = 0, FN = 200, TP = 0), illustrating that its ordering improved while its zero-threshold class calls did not. Additional quantitative checks: the mean `pc1_complex` was 0.9486 for GRH-true classes versus 0.9040 for GRH-false classes; the mean `pc1_mag` was 0.9818 for GRH-true versus 0.9707 for GRH-false. Leave-one-class-out class-level evaluation using the complex spectral features across all 11 classes yielded 11/11 correct classifications.

<challenges>
The main analytical challenge was incomplete artifact presence in the workspace. I could not use the named `.npz` coefficient archives for F9 and F10 because they were not present; instead, I regenerated those coefficients from scratch with PARI/GP through `cypari2`. This required package installation and increasing PARI stack memory to 4 GB. A second challenge was that the main feature CSV contains only |S_k| magnitudes, so the true complex M_jk matrix could not be reconstructed from that table alone; I therefore had to rebuild the missing complex S_k arrays for F9–F12 using the validated summation engine. A third issue is methodological: the spectral features are class-level constants replicated across 200 peaks within each class, so peak-level bootstrap confidence intervals do not reflect independent class-to-class uncertainty and therefore overstate precision. I report those CIs because they were requested and are directly comparable to prior peak-level analyses, but they should be interpreted cautiously. Finally, the known S_7 leak at N = 10^6 remains relevant for general-purpose peak-level classifiers, so I excluded S_7 from the baseline features to match the established strict protocol.
</challenges>
<discussion>
The analysis supports the research hypothesis. Phase-inclusive information in the true complex cross-term matrix contains discriminative structure that is largely absent from the magnitude-only proxy. In this dataset, the complex spectral features cleanly separated known GRH-true classes from known GRH-false classes, correctly generalized to the novel violator family F12, and avoided the important failure mode of incorrectly flagging F6 and F7 as violators. This directly addresses the main limitation identified in the prior magnitude-only spectral analysis. At the same time, the very strong result must be interpreted with appropriate caution: these spectral features are computed once per class and then copied to all peaks, so the effective sample size for the spectral-only comparison is the number of classes, not the number of peaks. Thus, the perfect AUC is a real finding for the present class set, but it is not yet sufficient to claim broad out-of-family generalization beyond these 11 classes. The combined model’s lower AUC than the spectral-only model suggests that adding noisy peak-level magnitude features can obscure an otherwise very clean class-level spectral signal. Scientifically, the results imply that phase relationships among the ω-stratified sums—not just their magnitudes—encode a robust signature of whether the underlying coefficient arithmetic corresponds to a GRH-true or GRH-false family.
</discussion>
